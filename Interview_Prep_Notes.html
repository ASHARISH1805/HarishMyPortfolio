Part 1: Interview Questions & Answers

1. Project Overview & Architecture

Q: Can you briefly explain the architecture of your portfolio application?

Key Points to Hit:

  • It is a Full-Stack Web Application.
  • Frontend: Vanilla HTML, CSS, and JavaScript for maximum performance and lightweight loading.
  • Backend: Node.js with Express.js REST API.
  • Database: PostgreSQL for structured data storage (Skills, Projects, Internships).
  • Admin Panel: A custom-built CMS (Content Management System) to manage portfolio content dynamically without redeploying code.

Q: Why did you choose Vanilla JavaScript instead of a framework like React or Vue for the frontend?

"I wanted to demonstrate that I can build complex, dynamic applications (like the Admin Dashboard with Drag-and-Drop) using core web technologies. For a portfolio, a framework might be overkill; this approach ensures the fastest possible load times (FCP - First Contentful Paint) and cleaner DOM manipulation."

2. Backend & API (Node.js & Express)

Q: How do you handle file uploads for project images and videos?

"I use the Multer middleware in Express. Files are uploaded to a local uploads/ directory on the server. I generated unique filenames using timestamps to prevent conflicts. The database stores the relative path to the file, and the frontend requests it via a static file route."

Q: Your application has an Admin Panel. How did you secure the API routes?

"I implemented a custom Middleware (authMiddleware in server.js). It checks for a secure token/password in the request headers. For the login itself, I integrated Google OAuth2 using the google-auth-library. The backend verifies the Google ID token and ensures the email is in my allowedEmails whitelist before granting access."

Q: How did you implement the 'Contact Me' feature?

"When a user submits the form, the data is first saved to the PostgreSQL messages table for persistence. Then, I use Nodemailer to trigger an asynchronous email notification to my personal inbox, ensuring I never miss an inquiry."

3. Database (PostgreSQL)

Q: How does your application handle database schema changes or migrations?

"I implemented a robust 'Auto-Migration' system (ensureSchema function). On server startup, the code checks if the necessary tables (Projects, Skills, etc.) exist and executes SQL CREATE TABLE IF NOT EXISTS or ALTER TABLE commands. This makes deployment seamlessâ€”I don't need to manually run SQL scripts on the server."

Q: Why PostgreSQL over MongoDB (NoSQL)?

"My portfolio data is highly structured (Projects have specific fields like title, tech stack, links). PostgreSQL's relational model enforces this consistency. It also allows me to perform complex queries easily, such as 'Count all projects where visibility is true', which I use for the Stats/Hero section."

4. Code Quality & Implementation Details

Q: I see you have a 'Micro-SaaS' section. How is that different from standard projects in your code?

"The Micro-SaaS section was a custom addition requiring specific metadata (Status, Role, Color Gradients). I created a dedicated table micro_saas and implemented specific backend logic to handle its unique coloring and status fields, showing my ability to model complex business entities."

Q: How does the Drag-and-Drop reordering work in the Admin Panel?

"I used Sortable.js on the frontend to allow drag-and-drop interactions. When an item is dropped, I capture the new order of IDs and send them to the /api/admin/reorder endpoint. The backend then executes a Transaction (BEGIN...COMMIT) to update the display_order field for all items safely, ensuring data integrity."

5. Behavioral & Problem Solving

Q: What was the most challenging bug you faced during development?

"I faced an issue where pushing large project demo videos (~60MB) caused Git HTTP 408 Timeouts. I reasoned that storing large binaries in Git is an anti-pattern. I solved it by removing heavy assets from version control, configuring Git buffer sizes, and planning for external Object Storage (like AWS S3) in a production environment."

Q: If you had to scale this application to 10,000 users, what would you change?

  • Database: Move from local PostgreSQL to a managed cloud database (AWS RDS).
  • Storage: Store user uploads in AWS S3 or Cloudinary.
  • Caching: Implement Redis to cache API responses (e.g., /api/projects) to reduce DB load for static content.

Part 2: Challenges Faced (Implementation Stories)

Use this to answer: "Tell me about a time you overcame a technical hurdle."

1. General Challenges (Process & Product)

Balancing "Showcase" vs. "Performance"
Challenge: High-quality demo videos and images slow down the website.
Solution: Optimized assets and implemented lazy loading. Learned that "speed" is a key feature.

Scope Creep (Feature Expansion)
Challenge: Project grew from static HTML to dynamic CMS to Micro-SaaS.
Solution: Learned Modular Architecture. Separated code (admin.js vs server.js) so new features didn't break old ones.

2. Technical Challenges (Code & Implementation)

Handling Large File Uploads & Git Limits
Challenge: Pushing ~60MB video caused HTTP 408 Timeouts.
Solution: Realized Git isn't for binaries. Configured buffer settings temporarily but planned for S3 Object Storage.

Cross-Platform File Paths
Challenge: Windows backslashes (\) broke image URLs in browsers.
Solution: Wrote a custom path normalization utility in Node.js to enforce forward slashes (/).

Google OAuth Integration
Challenge: Securing admin access without managing passwords.
Solution: Implemented backend token verification against a strict Email Whitelist.

3. Conceptual Challenges (Architecture)

Designing a "Generic" Admin System
Challenge: Avoid writing duplicate "Save" functions for 6 different sections.
Solution: Designed a Generic API Schema (/api/admin/save with table parameter) to handle dynamic SQL updates.

Database Schema Evolution
Challenge: Managing database changes (new columns) during development.
Solution: Wrote an Auto-Migration script (ensureSchema) to create tables/columns automatically on startup.
Summary Statement:
"The biggest takeaway was learning how to build a self-sustaining system. By overcoming challenges like database schema evolution and generic API design, I turned a simple website into a full Content Management System (CMS)."
